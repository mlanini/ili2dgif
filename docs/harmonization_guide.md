# Harmonization Guide for Heterogeneous Geodata towards DGIF V3

## Overview

This document describes best practices for harmonizing geodata from
heterogeneous sources (e.g. **swissTLM3D**, **Overture Maps**, **OSM**) into the
**DGIF V3** (INTERLIS) model.  The goal is to produce GeoPackage datasets
conformant to the DGIF V3 schema, semantically consistent regardless of the
original source.

---

## 1. Pipeline Architecture

Each data source follows the same three-stage schema:

```
[Source Data]  -->  [Mapping CSV]  -->  [ETL Transform & Load]  -->  [DGIF GPKG]
```

| Component | Purpose |
|---|---|
| **Mapping CSV** | Mapping table `source -> DGIF`: class, attribute, value |
| **ETL orchestrator** | Source file discovery, iteration, logging |
| **ETL transform** | OGR reading, CSV lookup, geometry transformation, sqlite3 INSERT |
| **DGIF GPKG** | GeoPackage generated by ili2gpkg with `--smart2Inheritance` |

## 2. Fundamental Harmonization Rules

### 2.1 DGIF Attribute Names

Column names in the GeoPackage are **lowercase** (generated by ili2gpkg).
The mapping CSV uses **camelCase** format (`railwayClass`, `waySignificance`)
which is then compared with `lower()` in the ETL.

**Mandatory check**: before adding an attribute to the CSV, verify that it
actually exists in the DGIF schema via:

```sql
-- Actual columns of the DGIF table
PRAGMA table_info(transportation_railway);

-- ILI -> SQL mapping
SELECT SqlName, IliName FROM T_ILI2DB_ATTRNAME
WHERE ColOwner = 'transportation_railway';
```

If an attribute in the CSV does not match any real column, the value is
**silently ignored** by the ETL (the `attr_lower in dgif_cols` check fails).

### 2.2 DGFCD Domain Values

Attribute values must come from the DGFCD catalogue
(`DGFCD_AttributeValueConcepts.xml`).  Examples:

| DGIF Attribute | Allowed Values |
|---|---|
| `railwayClass` | `branchLine`, `highSpeedRail`, `mainLine` |
| `railwayUse` | `tramway`, `funicular`, `undergroundRailway`, `railRapidTransit`, ... |
| `waySignificance` | `primaryWay`, `secondaryWay`, `tertiaryWay`, `localWay`, `minorWay`, ... |
| `meansTransportation` | `pedestrian`, `bicycle`, `automobile`, `train`, `tram`, ... |
| `waySurfaceCategory` | `hardPaved`, `loosePaved`, `unpaved` |

### 2.3 Geometry

The DGIF schema declares a geometry type for each class (POINT, LINESTRING,
POLYGON).  The pipelines must handle discrepancies:

| Case | Strategy |
|---|---|
| Source = LINESTRING, Target = LINESTRING | Direct reprojection |
| Source = POLYGON, Target = POINT | Centroid extraction |
| Source = POINT, Target = POLYGON | Do not insert (fallback not possible) |
| Source 3D (Z) | Flatten to 2D (`ogr.ForceTo(geom, ogr.wkbXxx)`) |

**Warning**: the geometry column in the GPKG is called `ageometry` (NOT `geometry`).
ili2gpkg renames `geometry` from the INTERLIS model to `ageometry` to avoid
conflicts with GeoPackage reserved columns.

### 2.4 Coordinate Reference System

DGIF V3 uses **WGS 84 (EPSG:4326)**.  The pipelines must reproject:

- swissTLM3D: LV95/LN02 (EPSG:2056) -> WGS 84
- Overture Maps: already in WGS 84, no reprojection needed
- OSM: already in WGS 84

### 2.5 Date Format (`beginlifespanversion`)

Always use **ISO 8601**: `YYYY-MM-DD` or `YYYY-MM-DDTHH:MM:SSZ`.

- swissTLM3D: the `datum_erstellung` field uses format `YYYY/MM/DD`
  -> normalize with `replace("/", "-")`
- Overture Maps: already uses ISO 8601 (`update_time` in GeoJSON)
- If the date is not available, use the current ETL run date

### 2.6 Unique Identifiers

| DGIF Field | Source |
|---|---|
| `T_Ili_Tid` | Source ID (UUID, TID, `@id`) or generated UUID |
| `uniqueuniversalentityidentifier` | Same value as `T_Ili_Tid` |
| `externalreferences` | Reference to the source (optional) |

## 3. Mapping CSV Structure

The CSV uses `;` as delimiter.  Columns:

| # | Column | Description |
|---|---|---|
| 0 | NO | Sequential number |
| 1-4 | Source | Theme/layer/attribute/value of the data source |
| 5 | Geometry | Source geometry type |
| 6 | Mapping Description | `OK` = direct mapping, `Generalization` = approximation |
| 7 | DGIF Feature Alpha | DGIF class name (camelCase) |
| 8 | DGIF Feature 531 | DGFCD class code |
| 9 | DGIF Attribute Alpha | First DGIF attribute (camelCase) |
| 10 | DGIF Attribute 531 | DGFCD attribute code |
| 11 | DGIF AttributeValue Alpha | First attribute value |
| 12 | DGIF Value 531 | DGFCD numeric value code |
| 13-16 | (same as 9-12) | Second DGIF attribute (optional) |

converge on the **same** DGIF attributes and values.

## 5. Common Mistakes to Avoid

| Mistake | Consequence | Prevention |
|---|---|---|
| Using DFDD attribute names (e.g. `gaugeOfRailway`, `functionalClass`) | Value silently lost | Validate against `PRAGMA table_info()` |
| Column `geometry` instead of `ageometry` | INSERT fails | Always use `ageometry` in checks |
| Dates in `YYYY/MM/DD` format | Inconsistency between sources | Normalize to ISO 8601 |
| Values outside DGFCD domain | Non-interoperable data | Consult `DGFCD_AttributeValueConcepts.xml` |
| Inserting `0`/`"unknown"` for NOT NULL | Indistinguishable from real data | Prefer `noInformation` / `-999999` |

## 6. Procedure for Adding a New Data Source

1. **Analyze the source schema**: themes, layers, attributes, geometry types
2. **Create the mapping CSV**: for each source combination, find the
   corresponding DGIF class in the DGFCD catalogue (`DGFCD_FeatureConcepts.xml`)
3. **Validate attributes**: every `DGIF Attribute Alpha` must correspond to a
   real column verified with `PRAGMA table_info()`
4. **Validate values**: every `DGIF AttributeValue Alpha` must come from
   `DGFCD_AttributeValueConcepts.xml`
5. **Implement the ETL**: copy the pattern from `etl_overture_transform.py`:
   - OGR reading
   - CSV lookup with 4-level fallback
   - Centroid/reprojection handling
   - INSERT via sqlite3
6. **Test**: run with a test dataset and verify:
   - No attributes silently lost (count successful INSERTs)
   - Valid geometries (check `gpkg_geometry_columns`)
   - Dates in ISO 8601 format
   - Correct domain values

## 7. Cross-Source Validation

To verify consistency between datasets produced from different sources:

```sql
-- Compare Railway attributes across sources
SELECT railwayclass, railwayuse, COUNT(*)
FROM transportation_railway
GROUP BY railwayclass, railwayuse;

-- Verify ISO 8601 dates
SELECT beginlifespanversion
FROM transportation_railway
WHERE beginlifespanversion NOT LIKE '____-__-__%';

-- Search for out-of-domain values
SELECT DISTINCT waysignificance
FROM transportation_landtransportationway
WHERE waysignificance NOT IN (
    'primaryWay','secondaryWay','tertiaryWay',
    'localWay','minorWay','intraUrbanAccessWay',
    'intraUrbanInterconnectionWay','noInformation'
);
```
